Every distributed dispatch system eventually hits the same bug: two workers grab the same job. I hit it in DeliverIQ while load-testing three API replicas against one orders table — occasionally, two replicas would claim the same order and a rider got assigned twice.

The naive claim looks innocent:

```sql
SELECT id FROM orders WHERE status = 'pending' LIMIT 1;
-- ...application decides...
UPDATE orders SET status = 'claimed', worker = $1 WHERE id = $2;
```

Between the `SELECT` and the `UPDATE`, another replica can run the same `SELECT` and see the same row. That gap is the race. You can shrink it, but you can't close it from the application side.

## The fix is one clause

```sql
BEGIN;
SELECT id FROM orders
WHERE status = 'pending'
ORDER BY priority DESC, created_at
LIMIT 10
FOR UPDATE SKIP LOCKED;
-- claim the rows, publish events
COMMIT;
```

`FOR UPDATE` locks the selected rows for the duration of the transaction. `SKIP LOCKED` is the magic: instead of *waiting* on rows another worker holds, the query skips them and returns the next unlocked rows. Each replica atomically gets a disjoint batch. No coordinator, no distributed lock service, no Redis `SETNX` dance.

## Why this beats the alternatives

- **Advisory locks** work but push lock bookkeeping into your application — you now own lock namespacing and release-on-crash semantics.
- **A dedicated queue (SQS, RabbitMQ)** decouples claiming from your data, which means a job can be delivered while its row says something else. With SKIP LOCKED, the claim and the state change commit *in the same transaction* — atomically, or not at all.
- **Crash safety comes free.** If a worker dies mid-claim, the transaction rolls back, the row lock evaporates, and the order is claimable again. Nobody has to notice.

## The trade-offs (write them down)

- Claims are bounded by transaction lifetime. Long-running work should claim, commit a `claimed` status, and process outside the transaction — or the row lock outlives its welcome.
- Locked rows are invisible to other workers' scans, so "queue depth" queries need `SKIP LOCKED` awareness too.
- Postgres is now on your hot path for both data *and* dispatch. At some throughput, that stops being free. Measure — for DeliverIQ, ~123 RPS with p99 of 220 ms was well inside comfortable territory.

After the change, the Locust suite that reliably reproduced the double dispatch ran clean: zero duplicate assignments under concurrent load.

The bigger lesson: the database you already run is often the most correct queue you can get. Reach for new infrastructure when it buys you something the transaction model can't — not before.
